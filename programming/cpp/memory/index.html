    Memory and Smart Ptr

    动态内存管理的问题1——内存泄漏

    众所周知，一个进程占有的内存包括代码段，数据段，栈内存和堆内存。其中堆内存可由程序动态申请，动态释放。Java和Python等语言存在垃圾回收机制，程序员不用太操心动态内存分配问题。然而对于C++而言，对动态内存使用不当可导致内存泄露。

    内存泄漏

    操作系统分配一块动态内存后，程序没有主动释放它，使得该块内存无法再次被操作系统分配。

    例如

    int *malloc() {
    int *p = new int;
    return p;
}

int main() {
    auto c = malloc();
}

    malloc函数内使用new操作符分配了一块内存，但是一直到main函数结束却都没有用delete释放它。这就是内存泄漏较为典型的例子。

    第一个智能指针类

    为了使每块被分配内存都保证释放一次，引入如下类：

    class IntPtr {
private:
    int *p;

public:
    IntPtr()
        : p{new int(0)} {
        cout << "malloc" << endl;
    }
    int &operator*() {
        return *p;
    }
    ~IntPtr() {
        cout << "free" << endl;
        delete p;
    }
};

int main() {
    IntPtr a;
    cout << *a << endl;
    *a = 3;
    cout << *a << endl;
}

    运行结果：

    malloc
0
3
free

    可以看到，在main函数退出时，IntPtr的析构函数被执行，释放了内存。因此使用智能指针类IntPtr能够避免内存泄漏问题。

    动态内存管理的问题2——重复释放

    有下列程序：

    int main() {
    IntPtr a;
    IntPtr b{a};
}

    运行结果：

    malloc
free
free
Segmantation fault!

    该程序中，变量a维护一个指针a.p，指向一块存有0的内存，变量b调用默认拷贝构造函数，复制a.p的值给b.p。但在main函数退出时，变量a和b都要调用各自的析构函数，它们的成员p是两个变量，但是指向同一块动态内存（因为值是拷贝过去的）。所以该块内存被释放了两次，导致出错了。

    我们不可能使用一个没有办法拷贝的指针。但一般的拷贝会导致错误，说明要引入新的拷贝——移动拷贝。

    为了不让IntPtr类型的变量维护的int型动态内存被多次释放，只能存在且仅存在唯一的int*类型变量指向它。

    如果拷贝IntPtr时不复制属性p的值，而是转移属性p的所有权，那么问题就能解决了。对于每块被动态分配的内存，永远只存在一个IntPtr实例，它的p属性指向这块内存。
