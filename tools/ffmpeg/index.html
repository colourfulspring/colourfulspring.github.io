    ffmpeg

    This note is mainly about a famous tool to handle videos and audios. As its official website introduces, ffmpeg is a solution to record, convert and stream audio and video. In my opinion, this tool has many advantages, like open-source, which means no ads, cross-platform, which satisfies various requirements. Thus, I highly recommend this tool to everyone.

    Basic Understanding

    It is known to all that a movie (or sound) file usually consists of video streams, audio streams and subtitles, etc. Each of them is a sequence of frames and has its own encoding and decoding methods. Apart from data conding, sometimes we want to apply some changes to data contents, such as enlarging frame size, changing fps, etc. This explains the fact that the ffmpeg has five inner parts:

    • demuxer: split the input file(s) into streams
    • decoder: decode a stream to raw data(frames).
    • filter: modify data contents
    • encoder: encode data(frames) to a stream
    • muxer: combine streams to an output file.

    As long as you use ffmpeg, your handing process will include some or all of these five parts in order.

    Documentation

    The transcoding process in ffmpeg for each output can be described by the following diagram:

     _______              ______________
|       |            |              |
| input |  demuxer   | encoded data |   decoder
| file  | ---------> | packets      | -----+
|_______|            |______________|      |
                                           v
                                       _________
                                      |         |
                                      | decoded |
                                      | frames  |
                                      |_________|
                                           |
                                           (filter)
                                           |
                                           v
                                       _________
                                      |         |
                                      | decoded |
                                      | frames  |
                                      |_________|
 ________             ______________       |
|        |           |              |      |
| output | <-------- | encoded data | <----+
| file   |   muxer   | packets      |   encoder
|________|           |______________|

    ffmpeg calls the libavformat library (containing demuxers) to read input files and get packets containing encoded data from them. When there are multiple input files, ffmpeg tries to keep them synchronized by tracking lowest timestamp on any active input stream.

    Encoded packets are then passed to the decoder (unless streamcopy is selected for the stream, see further for a description). The decoder produces uncompressed frames (raw video/PCM audio/...) which can be processed further by filtering (see next section). After filtering, the frames are passed to the encoder, which encodes them and outputs encoded packets. Finally those are passed to the muxer, which writes the encoded packets to the output file.

    PCM

    Pulse-code modulation (PCM) is a method used to digitally represent sampled analog signals. It is the standard form of digital audio in computers, compact discs, digital telephony and other digital audio applications. In a PCM stream, the amplitude of the analog signal is sampled regularly at uniform intervals, and each sample is quantized to the nearest value within a range of digital steps.

    For muxer and demuxer, their implementations are simple. Maybe ffmpeg itself implement them. We just need to give it some arguments.

    However, encoder and decoder are different. For example, some rich man want to see high resolution movie but don't care file size, so they need avi format and matching coders, while others care file size more because of storage expense, so mp4, mkv format and lossy compression algorithm like H264, H265 and AAC are suitable for them.     Filter

    Before encoding, ffmpeg can process raw audio and video frames using filters from the libavfilter library. Several chained filters form a filter graph. ffmpeg distinguishes between two types of filtergraphs: simple and complex. After them, we will introduce many filters which are often used.

    Simple Filtergraphs

    Simple filtergraphs is just a sequence of filters. The output of the former one is treated as the input of the latter one. We use -filter option with -vf and -af aliases for video and audio respectively.

    Complex Filtergraphs

    Complex filtergraphs are those which cannot be described as simply a linear processing chain applied to one stream.

    crop

    Crop the input video or image.

    Option Description
    -vf crop=w:h:x:y Crop area with size w*h at position (x,y)

    scale

    Resize the input video or image.

    Option Description
    scale=w:h resize the input to a size of w*h

    Muxer and Demuxer

    As mentioned before, muxer and demuxer are tools to handle streams. To finish this work, a -map option is provided together with stream specifiers.

    Stream Specifier

    Use -map as an example to tell stream specifier

    Option Description
    -map 0:a:1 input file 0, audio stream 1

    Encoder and Decoder

    Select an encoder (when used before an output file) or a decoder (when used before an input file) for one or more streams. codec is the name of a decoder/encoder or a special value copy (output only) to indicate that the stream is not to be re-encoded.

    -codec:v, -codec:a or their abbrevation -c:v, -c:a means encoders and decoders.

    NVIDIA Support

    Stream Copy

    Stream copy is a mode selected by supplying the copy parameter to the-codec option. It makes ffmpeg omit the decoding and encoding step for the specified stream, so it does only demuxing and muxing. Since there is no decoding or encoding, it is very fast and there is no quality loss. However, it might not work in some cases because of many factors. Applying filters is obviously also impossible, since filters work on uncompressed data.

    Option Description
    -c:v:1 libx264 use libx264(CPU) to encoded video stream 1
    -c:a copy copy audio stream
    -c:v:1 h264_cuvid use h264_cuvid(GPU) to encoded video stream 1
